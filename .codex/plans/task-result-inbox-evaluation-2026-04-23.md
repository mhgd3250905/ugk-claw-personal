# 后台任务结果页方案评估

- 日期：2026-04-23
- 状态：评估完成，待用户确认是否进入实现
- 结论：建议做，但不要新造一套独立聊天系统；应基于现有全局活动能力升级为“任务消息 / 结果收件箱”

## 1. 背景问题

当前后台任务完成后需要把结果投递到指定会话，这个模型和真实使用习惯冲突：

1. 用户经常切换会话，任务完成时未必还停留在原会话。
2. 后台任务结果属于“异步通知 + 可追溯结果”，不天然属于当前打开中的聊天上下文。
3. 为了把结果塞回某个会话，前后端都要维护额外的目标会话绑定逻辑，复杂且脆弱。

## 2. 现状核对

现有实现并不是零起步，已经有可复用地基：

1. `src/agent/agent-activity-store.ts`
   - 已有全局活动存储
   - 已支持 `runId`、`conversationId`、`readAt`、标题、正文、文件等字段
2. `src/routes/activity.ts`
   - 已有 `GET /v1/activity`
   - 已有 `POST /v1/activity/:activityId/read`
3. `src/ui/playground-conn-activity-controller.ts`
   - 已有活动列表加载能力
   - 已有“查看过程 / run 详情”的打开逻辑
4. `docs/playground-current.md`
   - 已存在顶部通知流、活动列表、toast 等概念

结论：后端持久化、已读状态、过程查看这些核心能力基本已经有了，缺的是产品层面的收口。

## 3. 推荐方案

### 方案定位

把“后台任务结果页”定义为：

- 一个第一优先级的**任务消息收件箱**
- 展示的是后台任务完成后的结果消息
- 视觉可以像 chat
- 但语义上不是第二套会话，不参与会话上下文

### 推荐形态

1. 顶部状态栏新增 `任务消息` 按钮
2. 按钮显示未读红点 / 数字
3. 点击进入独立页面或全屏级视图
4. 页面内按时间倒序展示后台任务结果
5. 每条消息底部提供：
   - `任务ID`：点击复制
   - `复制`：复制结果正文
   - `查看过程`：打开现有 run 过程详情

## 4. 不推荐的做法

### 不推荐 A：继续把结果强行发回指定会话

这个模型已经和用户行为打架，继续补就是缝缝补补，后面还得继续解释“为什么消息跑错地方”。

### 不推荐 B：新建一套完全独立的消息表和接口

现在活动存储已经存在，再造一套等于重复建设，后面 unread、详情、分页、去重、状态同步全都要再做一遍，纯属给自己找事。

### 不推荐 C：把任务结果页做成“什么都能发”的第二聊天页

一旦它变成第二套聊天系统，用户会分不清：

- 任务消息
- 会话消息
- 系统通知
- 全局活动

边界会迅速失控。

## 5. 推荐的数据与交互语义

### 任务消息的来源

只收口“后台任务完成后的结果消息”，包括：

1. 成功
2. 失败
3. 部分成功 / 需要人工继续处理

### 任务ID定义

需要尽快统一：

1. 用户看到的“任务ID”建议优先使用 `runId`
2. `sourceId` 更像业务源对象 ID，不适合直接当任务 ID 文案

如果两者都重要，UI 上主展示 `任务ID = runId`，其他源信息放副标题。

### 已读逻辑

建议：

1. 以服务端 `readAt` 作为唯一真源
2. 进入页面不要直接全部已读
3. 用户展开或明确浏览某条后再标已读，或者提供“全部已读”

否则红点会像假的一样，刚点开就没了，体验很虚。

## 6. 对现有系统的影响评估

### 直接影响

1. 后台任务结果不再依赖“投递到当前 / 指定会话”
2. 会话页职责收敛回“当前对话上下文”
3. 活动页升级为正式结果入口

### 间接影响

1. 通知流与未读计数要统一口径
2. 结果页与会话页之间需要可跳转但不能强绑定
3. conn run 详情弹层需要支持从结果页进入

### 数据兼容性

当前活动表已有 `runId`、`conversationId`、`readAt`，兼容性风险低。

## 7. 最小实现建议

### 第一阶段

1. 顶栏增加 `任务消息`
2. 用现有 `GET /v1/activity` 拉取列表
3. 用现有 `readAt` 驱动未读角标
4. 结果项复用 chat 风格消息壳
5. 复用已有“复制正文”“查看过程”

### 第二阶段

1. 增加筛选：成功 / 失败 / 未读
2. 增加“打开原会话”
3. 增加“全部已读”
4. 增加分页或无限滚动

## 8. 最终建议

建议立项，方向是对的。

但实现上不要说“新建任务结果页面”就另起炉灶，正确做法是：

**把现有 activity 体系产品化为一个正式的“任务消息 / 结果收件箱”页面，并把后台任务结果从会话投递中解耦出来。**

## 9. 待用户确认

如果进入实现，建议下一步先补一份正式实施计划，明确：

1. 页面是独立路由，还是在 `playground` 内切换视图
2. `任务ID` 是否统一使用 `runId`
3. 是否在第一版就支持“全部已读”

## 10. 用户已确认的产品边界

截至 2026-04-23，用户已明确确认：

1. 采用**独立任务消息页**
2. 页面视觉可像 chat，但语义不是会话
3. 后台任务结果**不再以回投原会话为主路径**
4. **不需要**提供“打开原会话”

因此后续实现默认按以下口径推进：

1. 任务消息页是后台结果的一等入口
2. 结果项只服务于查看结果、复制结果、查看过程、复制任务 ID
3. 会话页职责回到普通对话，不承担后台结果归档职责
